1. How does the base get its cm-quality location?
There are really two cases.
	•	If you use WISCORS/NTRIP, then your device operates as a rover and gets centimeter-grade coordinates from the network corrections.  
	•	If WISCORS is unavailable, one of your identical devices can be switched into local base mode and either:
	•	sit on a known surveyed control point, which is the proper way to preserve true survey coordinates, or
	•	run survey-in, which lets the base average its own position and then broadcast RTCM corrections. That gives the rover very good relative centimeter accuracy, but the whole job may be shifted if the base’s absolute coordinate is not truly known.  

For staking irons to an existing coordinate system, the best practice is still:
known control point or WISCORS/NTRIP.
A surveyed-in temporary base is a good fallback for local work, but not ideal when legal coordinate fidelity matters.  

2. Do you need the moving-base feature?
Probably no.
Moving base is intended for cases where both receivers are moving and you want the relative vector or heading between them, such as vehicles, robots, UAVs, or attitude sensing. Your surveying use case is a static base + moving rover, or just a rover on WISCORS, so standard RTK is the right mode.  

3. What is the purpose of the cameras on some expensive versions?
Those are typically for usability and documentation, not for the RTK solution itself. They are used for:
	•	visual or AR-style stakeout,
	•	point photo documentation,
	•	site context capture,
	•	sometimes assisting the user in positioning the pole over a target.

They are helpful, but they are not necessary for centimeter GNSS stakeout. This is more of a premium workflow feature than a core positioning requirement. The underlying ZED-F9P functionality is GNSS/RTK, not camera-driven.  

4. What kind of I/O do the stations require?
The ZED-F9P family provides the usual embedded interfaces: UART, USB, SPI, and I2C, plus timing outputs. In practice, for your design, the important I/O is:
	•	GNSS antenna input,
	•	power,
	•	serial/USB connection to your host controller,
	•	correction-data path in and/or out,
	•	optional radio, Bluetooth, Wi-Fi, or cellular interface.  

For a dual-purpose identical device:
	•	in rover mode, the unit receives RTCM corrections from WISCORS/NTRIP or from your other unit,
	•	in base mode, it outputs RTCM corrections over radio, serial, Bluetooth bridge, or an NTRIP caster/client path.  

5. How complex are the local functions, or does most control happen in the app/web interface?
Most of the complexity should live outside the GNSS module. The F9P is best treated as a positioning engine. Your own app or controller should handle:
	•	entering target coordinates,
	•	switching between base and rover profiles,
	•	showing fix status,
	•	stakeout guidance,
	•	logging and export,
	•	radio/network setup.

The receiver itself is configured once and then mostly streams status and position. u-blox and ArduSimple both expose base and rover setup largely through configuration profiles rather than requiring lots of on-device logic.  

A good architecture is:
	•	same hardware on both units
	•	role selected in software
	•	a simple local screen for status
	•	most workflow handled in the stakeout app

6. Since these are battery powered, how much effort should go into battery life?
Enough to matter, but not so much that it compromises reliability.

For surveying, uptime and fix quality matter more than squeezing every last milliamp. I would optimize battery life in a practical way:
	•	efficient display,
	•	radios that can be disabled when not needed,
	•	clean power design,
	•	sleep mode for idle periods,
	•	battery sized for a field session.

Because you want each unit to be either a base or a rover, the “base” unit may sit for hours broadcasting corrections, so you do need a decent battery budget. But the right answer is usually bigger battery + solid power design, not extreme micro-optimization. This is especially true if one unit may become a field base where continuous operation is more important than tiny power savings. The F9P is specifically aimed at embedded centimeter-accuracy applications, so battery operation is a common design pattern.  

7. How important are multi-band L1/L2 or L1/L5?
For centimeter RTK surveying, very important.
The ZED-F9P’s core value is that it is a multi-band RTK receiver, which is what gives faster convergence and more reliable centimeter-level performance than cheap single-band GPS.  

The safest path is still the L1/L2 ZED-F9P family such as 02B/04B/05B, because that is the better-supported ecosystem for hobbyist and embedded survey builds. The L1/L5 branch exists, but for your first identical-base/rover design, L1/L2 is the simpler and more proven choice.  

8. Could you test most functionality with a cheaper GPS module?
You could test only the non-RTK parts:
	•	UI,
	•	menus,
	•	coordinate entry,
	•	map display,
	•	general software plumbing.

But you would not be testing the most important parts:
	•	RTCM handling,
	•	base/rover behavior,
	•	survey-in,
	•	fix status transitions,
	•	centimeter stakeout performance.

Those are exactly the reasons to use the ZED-F9P in the first place. So a cheap GPS is fine for interface mockups, but it is not a real substitute for development of your surveying workflow.  

9. Is the ZED-F9P too extensive for this project?
No. It is actually a strong fit.
It already supports the things your identical-unit concept needs:
	•	rover mode,
	•	base mode,
	•	RTCM correction output/input,
	•	survey-in,
	•	multi-band RTK,
	•	optional moving-base support if you ever wanted it later.  

You do not need to use all of those features on day one. But they give you room to start simple and expand later.

10. Which version do I think is best for this project?
For a new project with two identical interchangeable base/rover devices, I would still stay in the L1/L2 ZED-F9P line. A cheap 02B board is fine for development and proving the workflow. For a more final design, 04B or 05B are the better long-term choices if pricing and availability are close.

How do you setup the base to precisely 'know' its position?

Setting up a base station to "know" its exact position is the most critical step in an RTK workflow. Any error in the base station’s coordinates will be passed directly to the rover.
There are four primary ways to establish this "Ground Truth":

	1. The "Known Point" Method (Gold Standard)The most accurate method is to place the base station directly over a pre-existing survey monument or benchmark.Process: You physically center the base on a tripod over the marker.Data Entry: You manually enter the marker's certified coordinates (Latitude, Longitude, and Height) into your base station software.Antenna Height: You must precisely measure the vertical distance from the ground marker to the receiver's antenna reference point to ensure accurate elevation.

	2. Precise Point Positioning (PPP)If no survey marker exists, you can use PPP to find a high-accuracy coordinate from scratch.Recording: You set the base in a stationary position and record raw satellite data for several hours (ideally 4–24 hours).Post-Processing: You upload this data to an online service like the National Geodetic Survey's OPUS or the Canadian CSRS-PPP.Result: These services use global networks of reference stations to calculate your exact location within roughly 1–2 centimeters.
	
	3. "Survey-In" or Averaging (Fast & Simple)This is the most common method for day-to-day work where absolute global accuracy is less critical than relative accuracy between the base and rover.Process: The base sits for 5–15 minutes and averages its autonomous (non-corrected) position.Accuracy: This yields a coordinate accurate to about 1–3 meters globally, but the rover will still move with 1–2 centimeter precision relative to that base.Consistency: If you return to the same site later, you must use the exact same averaged coordinate you used the first day to keep your maps aligned.
	
	4. NTRIP Network InitializationIf you have internet access, your "local" base station can act as a rover for a few minutes.Process: You connect your base station to a regional CORS (Continuously Operating Reference Station) network via NTRIP.Initialization: The base receives corrections from the network to "Fix" its own position to centimeter accuracy. Once its position is fixed, you disconnect the internet and start broadcasting as a base station for your rovers.MethodTime RequiredAbsolute Global AccuracyBest Use CaseKnown PointMinutes1–2 cmConstruction sites with existing markersPPP4–24 Hours1–2 cmRemote wilderness or establishing new controlNTRIP5 Minutes2–5 cmUrban areas with cellular coverageSurvey-In5–15 Minutes1–3 MetersJobs requiring only relative accuracyAre you planning to set up a permanent base station or a temporary one for daily use?
